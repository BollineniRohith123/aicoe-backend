UC-004

Delete a Task

The user permanently removes a task from their list. To prevent accidental data loss, this action should be easily accessible but may include a confirmation mechanism, a best practice for user experience.

High Priority User Interaction Destructive Action

Actors

User

Preconditions

The user is viewing the task list (UC-001) with at least one task present.

Postconditions

The task is permanently removed from the database.
The task is no longer visible in the user's task list.

Business Value

Allows users to maintain a clean and relevant task list by removing obsolete or completed items. This is a critical maintenance feature for long-term usability.

1

The user locates a task they wish to delete.

2

The user clicks a 'delete' icon (e.g., a trash can) next to the task.

3

The system sends an asynchronous DELETE request to the backend API endpoint for that specific task (e.g., /api/tasks/:id).

4

The backend removes the task from the database and returns a success response (e.g., HTTP 204 No Content).

5

Upon receiving the success response, the system removes the task element from the UI with a smooth animation.

Deletion with Confirmation (Enhanced Flow)

This flow can be implemented if the risk of accidental deletion is deemed high.

2a

The user clicks the 'delete' icon.

3a

The system displays a confirmation modal or dialog: 'Are you sure you want to delete this task?' with 'Cancel' and 'Delete' buttons.

4a

The user clicks 'Delete'.

5a

The system proceeds with the DELETE request to the backend as in the main flow.

6a

If the user clicks 'Cancel', the modal closes, and no further action is taken.

Backend Deletion Failure
3a

The backend fails to delete the task (e.g., task not found, database error).

4a

The backend returns an error response.

5a

The system displays an error message: 'Could not delete the task. Please try again.' and the task remains visible in the list.
