UC-002

Add a New Task

Actor: User
High Priority
Page 2 of 5

Description

The user creates a new task to add to their list. The process is designed for speed and minimal friction, following the trend of intuitive and fast interaction seen in competitors like Any.do and Google Tasks.

Actors

User

Preconditions

The user is viewing the task list (UC-001).

Main Flow

The user clicks or focuses on the 'Add a new task' input field.
The user types the task description.
The user presses the 'Enter' key or clicks the 'Add' button.
The system performs client-side validation to ensure the task description is not empty.
The system sends an asynchronous POST request to the backend API endpoint (e.g., /api/tasks) with the new task data.
The backend receives the request, validates the data, saves the new task to the database, and returns a success response with the newly created task object (including its unique ID).
The system updates the UI by adding the new task to the top of the task list without a full page reload.
The 'Add a new task' input field is cleared, ready for the next entry.

Alternate Flows

Empty Task Submission
3a. The user presses 'Enter' or clicks 'Add' with an empty input field.
4a. The system's client-side validation fails.
5a. The system displays an inline error message (e.g., 'Task description cannot be empty') and prevents the API request from being sent.
Backend Save Failure
6a. The backend fails to save the task (e.g., database error).
7a. The backend returns an error response (e.g., HTTP 500).
8a. The system displays a user-friendly error message: 'Failed to add task. Please try again.'

Postconditions

The new task is successfully saved in the database and appears in the user's task list.
The application state is updated to reflect the new list of tasks.

Business Value

This is a core functional requirement that enables the primary purpose of the application. An efficient and reliable add feature is critical for user adoption and productivity.
